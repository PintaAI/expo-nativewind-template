## Expo Router / Native Tabs Notes

- `presentation: "formSheet"` must be configured on a screen in a parent/root `Stack` above the native tabs when it should overlay tab content.
- Do not put `formSheet` only on the first screen of a nested tab Stack. The first screen in a Stack renders as a normal screen because there is no underlying screen to present over.
- To make a form sheet visually/behaviorally match direct cashflow forms such as `entry-form`, `transfer`, or `quick-fill`, the route must be a direct child screen of the parent cashflow `Stack` using the shared `formSheetOptions`. Do not wrap that form in its own nested `_layout.tsx`/inner `Stack`; the extra native navigator layer changes header background, scene background, measurement, and detent behavior. If a folder route like `forms/wallet/index.tsx` needs matching behavior, delete `forms/wallet/_layout.tsx` and register both `forms/wallet` and any detail route such as `forms/wallet/detail` directly in `src/app/(cashflow)/_layout.tsx` with `formSheetOptions`.
- For a native tab bar action button that opens a root form sheet, keep the real form sheet route outside the tab group (for example `src/app/entry-form.tsx`) and register it in `src/app/_layout.tsx` with `presentation: "formSheet"`.
- `NativeTabs.Trigger` only appears for a real child route in the native-tabs layout. If the button is only a trigger, add a tiny internal route such as `src/app/(cashflow)/entry-form-trigger.tsx`, point the trigger `name` to that route, set `disabled`, and open the real form sheet from `listeners.tabPress` with `router.push("/entry-form")`.
- Native tab `tabPress` events in this Expo Router version are not preventable with `event.preventDefault()`. Use `disabled` on the trigger to block native tab selection while still receiving `tabPress`.
- For profile/account detail sheets under a tab stack, register the route in that tab's `_layout.tsx` with the same form sheet options used by cashflow forms: `presentation: "formSheet"`, `headerLargeTitle: false`, `headerTransparent: true`, `sheetAllowedDetents: "fitToContents"`, `sheetExpandsWhenScrolledToEdge: false`, and `sheetGrabberVisible: true`.
- Style custom form sheet bodies like `src/app/(cashflow)/forms/entry-form.tsx`: use a themed `ScrollView` (`bg-[--app-color-background]`, `contentInsetAdjustmentBehavior="automatic"`, padded content) and rounded bordered React Native surfaces instead of `@expo/ui/swift-ui` `Form` rows when the sheet needs custom visual layout.
- When a form sheet uses native `@expo/ui/swift-ui` controls such as `Slider`, prefer the `src/app/(cashflow)/(tabs)/profile/font-settings.tsx` pattern: keep `sheetAllowedDetents: "fitToContents"`, avoid explicit background colors on the outer React Native `View`/`Host`, wrap SwiftUI content in `Host matchContents={{ vertical: true }}` with `style={{ width: "100%" }}`, and use a padded `VStack` so the sheet height follows its content.

## Design Mindset

- Favor lightweight, integrated screens over decorative dashboard cards.
- Let native navigation own page titles and headers; avoid duplicating page headings in content unless the screen needs a distinct in-flow title.
- Lead with the data or primary action. Remove helper copy, subtitles, and wrappers when they do not add meaning.
- For cashflow screens, keep summaries direct and compact: label, privacy/action controls, key amounts, then optional expandable detail.
- Use visual containers for secondary/detail content, not for every major block. The main content should feel like part of the screen, not a card placed on top of it.
- Preserve whitespace and hierarchy through typography, spacing, and subtle theme surfaces instead of heavy borders, shadows, or decorative framing.

## EAS Update Flow

- Before publishing an OTA update, inspect `git status --short` and confirm whether the current dirty/untracked working tree should be shipped. EAS Update exports the current local files, not only committed changes.
- Use the globally installed `eas` binary in this workspace. `npx eas` may fail because `eas-cli` is not a local dependency.
- In non-interactive mode, include both `--channel` and `--environment`; EAS requires the environment flag.
- Production all-platform command:
  `CI=1 eas update --channel production --environment production --message "<message>" --non-interactive`
- If `--platform all` fails because the web export cannot resolve `expo-sqlite`'s `wa-sqlite.wasm`, publish native platforms separately:
  `CI=1 eas update --channel production --environment production --platform android --message "<message>" --non-interactive`
  `CI=1 eas update --channel production --environment production --platform ios --message "<message>" --non-interactive`
- `--non-interactive` still prints an Expo CLI warning; setting `CI=1` is the important part for export automation.
- EAS Update only ships JS/assets to builds with the same runtime version. Because `runtimeVersion.policy = appVersion`, native/config/dependency changes usually require a new EAS build instead.
- Record the Android/iOS update group dashboard URLs in the final response after publishing.

## Common Commands

- `npm run init-template` initializes a new app from the template.
- `npm run dev` starts the Expo dev server in dev-client mode with tunnel and cache clear.
- `npm run android` starts dev-client mode with tunnel and opens Android.
- `npm run ios` starts dev-client mode with tunnel and opens iOS.
- `npm run build:android` builds a local Android dev-client APK.
- `npm run build:android:preview` builds a local Android preview APK.
- `npm run build:ios` starts an EAS cloud iOS development build.
- `npm run run:android` uses `expo run:android` for local native Android install/run.
- `npm run web` starts web-only Expo dev server.
- `npm run lint` runs `expo lint`.

## NativeWind / Tailwind v4 Setup

- **Pipeline:** `postcss.config.mjs` -> `@tailwindcss/postcss` | `metro.config.js` -> `withNativewind(config)` | `nativewind-env.d.ts` -> `react-native-css/types`
- **No `babel.config.js`:** Expo SDK 56 handles Babel internally via Expo Router entry.
- **No `tailwind.config.*`:** Tailwind v4 uses CSS-based config via `@theme` directives.
- Theme tokens live in `src/global.css` and runtime values are provided by `VariableContextProvider` in `src/components/ThemeProvider.tsx`.
